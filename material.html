Beyond the Course Material

Several links to research literature are listed below. Just to be clear, this is not exam material. Just for your curiosity.
  • Automated testing

    One of the leading automated testing techniques is so-called search-based test (SBT), where you use search algorithms like evolutionary algorithms to find test inputs that would maximize your coverage. A standard starting literature for SBT is the survey paper by McMinn. It is a bit old by now, but it is still a good starting point. Many of the more modern work on automated testing uses this idea of search-based.

    McMinn, Search‐based software test data generation: a survey. In the Journal of Software testing, Verification and reliability, 14(2), 2004.

    Complementary to SBT is the symbolic execution approach, closely related to the verification approach from your 2nd part of the course. The symbolic approach exploits full semantic of the program, whereas with SBT we don't do this. Each approach (SBT vs symbolic) has its own strength and weakness.

    Baldoni et al. A survey of symbolic execution techniques. ACM Computing Surveys (CSUR), 51(3), 2018.

  • Proof asssitants To really produce formal proofs for larger problems you would want to use a proof assistant tool. Such a tool contains an implementation of logic rules. When you write a proof, it will be checked by the tool and the proof is really according to the rules. Most proof assistants also include a language layer that allows you to write proofs more abstractly (but still have the generated low level proof checked), and they often also have automation libraries that can do automated proving for certain types of problems.

    Some popular proof assistants: ROCQ, Agda, Isabelle, HOL4.

  • Theorem Provers A theorem prover can automatically prove a formula. Of course in general this is an undecidable proble, so every prover has its own range of decidability. Such a tool is typially not used to assist you in writing proofs. Rather, it is used as a backend for e.g. augomated progam verification or testing tools.

    They are usually implemented as a model finder ('SAT' solver). For example, given a formula p ∧ (q ⇒ r) ⇒ r a SAT solver tries to find an instance of value assignment for p,q,r that would make the formula true. if so, the formula is said to be satisfiable. For example, for that example formula p=true,q=true,r=true would make the formula true. So, the formula is satisfiable.

    If you want to know if the formula is valid (true for all value assignments to its free variables), we can check whether its negation, so: ¬(p ∧ (q ⇒ r) ⇒ r), has a model (is satisfiable). If it does, the formula has a counter example; so it is not valid. If it does not, then the formula is valid.

    Some popular theorem provers: CVC and Z3.
