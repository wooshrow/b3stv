List of common mistakes

Proof Assignment PART-2 common mistakes
  1. [XC] Wrong: an inference rule is applied in a context.

    When you have an inference rule: 

          φ
   --------
      ψ
    Note that inference rule works one direction! You use the rule above to derive ψ when you have inferred φ. You need to have this φ as a separate formula, and NOT in the context of another formula. So, the following examples are all INCORRECT use of the rule:
    • Inferring ¬ψ from ¬φ
    • Inferring ψ=f from φ=f
    • Inferring (∀ i : ψ : Q) from (∀ i : φ : Q)
  2. [TUI] Wrong: deductive inference that is technically useless.

    Consider this proof: 

    PROOF Termination Condition 1
[A1:] 0 ≤ i ≤ N  // invariant
[A2:] ... // some other part of the invariant
[A3:] i < N // loop guard
[G:] wp (C:=m ; found := found ∨ a[i]=x ; i := i+1) (m < C) 
BEGIN
   1. { calculating the goal G } n - (i+1) < n-1  
      // PROBLEM: this is a deductive proof. If you write this, it means you claim that 
      // n - (i+1) < n-1 is a valid conclusion. Buy you have NOT gotten so far yet. All
      // you mean to say is that goal G is *equivalent* to n - (i+1) < n-1, which is
      // quite different than saying that n - (i+1) < n-1 is valid!
      
   2. { rewrite 1}  n - i - 1 < n-1  
      // the same issue as above
      
   3. { rewrite 2 } -1 < 0  ... which is trivially true, so proof done!
      // PROBLEM: well, technically we just now derive a fact that is already known. In a deductive 
      // proof, it is not very useful to infer the formula 3 above. Similarly, inferring "true" is
      // also not very useful. Of course I understand that the author of this proof
      // means to say that 1 can be rewritten to 3, and therefore 1 is a valid conclusion.
      // But then we should not set up the proof the way it is now, which is technically wrong.
   
END
    Here is how we should set up the arguments properly: 
    PROOF Termination Condition 1
[A1:] 0 ≤ i ≤ N  // invariant
[A2:] ... // some other part of the invariant
[A3:] i < N // loop guard
[G:] wp (C:=m ; found := found ∨ a[i]=x ; i := i+1) (m < C) 
BEGIN
   1. { calculating the goal G } G  =  n - (i+1) < n-1  (we treat "G" as a name)
   2. { rewrite 1}  G  =  n - i - 1 < n-1     
   3. { rewrite 2 } G  =  -1 < 0  
   4. { 1 < 0 always hold } G  
END
Proof Assignment PART-1 common mistakes
  1. [TMR] Wrong: R-typed termination metric.

    Someone came up with a termination metric that maps to R (real numbers). This is problematical because you can't exclude Zenonian paradox. Similarly, don't map to Q; (rational numbers). For this reason, we choose to map to integers.

  2. [UDM1] Unjustified domain merge.

    Given a quantified formula, e.g.

    (1) (∀i : 0≤i < n : b[i])
    you want to rewrite (that is, transforming it to something else equivalent) it to:
    (2) (∀i : 0≤i < j : b[i]) ∧ (∀i : j≤ i < n : b[i])
    , based on the Domain Split theorem. Note that (1) and (2) are only equivalent if
    (3) 0≤ j < n
    If you don't have (3) then obviously it is not correct to do the rewrite.

    To show you what went wrong, here is the sequence of the rewrites, in smaller steps:

    (1) (∀i : 0≤i < n : b[i])
    = // (* wrong step! *) split 0≤i < n to 0≤i < j ∨ j≤i < n, based on Domain Merging thm A.4.16
    (∀i :0≤i < j ∨ j≤i < n : b[i])
    = // Domain Split (thm A.4.12)
    (2) (∀i : 0≤i < j : b[i]) ∧ (∀i : j≤i < n : b[i])
    The problematic step is marked above. It uses the theorem Domain Merging A.4.16, but notice that the step requires condition (3). If you have it, it's fine. If you don't then the step is not justified.

  4. [NFAL] No free ∀-lunch

    Someone wrote this proof: 

    PROOF main
[A1] ¬(∀ i: 0≤ i < n : b[i]) 
[A2] (∀ i: j≤ i < n : b[i])    
[G:] ¬(∀ i: 0≤ i < j : b[i]) 
BEGIN
(1) { follows from A2} j<n
...
    Well, step (1) is of course incorrect. If the domain of ∀ in A2 is empty, we can't have j<n, and so far we don't know whether this domain is empty or not.

  5. [IUA1] Incorrect use of [ANY x]

    You get the [SOME x] flag when you do ∃-elimination; this can appear in any line in your proof. On the other hand, the [ANY x] flag can only appear at the beginning of a (sub-) proof. In addition to introducing a scope, [ANY x] declares your intent to generalize the conclusion of your proof to (∀x ...).
    You obtain [SOME x] when you do ∃-elimination. See p24 of the slide predlogic and LN Section 3.7.2.
    For more on [ANY x], see LN Section 3.71. If you are impatient, p27 of predlogic shows an example (though you'll need the explanation in the LN to grasp it).

    Ok, here is an example of incorrect introduction of [ANY x], taken from what some of you did: 

    PROOF main
[A1] ¬(∀ i: 0≤ i < n : b[i]) 
[A2] (∀ i: j≤ i < n : b[i])    
[G:] ¬(∀ i: 0≤ i < j : b[i]) 
BEGIN
(1) { see the proof below } (∀ i: 0≤ i < j : b[i]) ⇒ false
       PROOF sub
       [A1:] (∀ i: 0≤ i < j : b[i])
       [G:] false
       BEGIN
       (1) {rewrite main.A1 with Negate-∀ thm} (∃ i: j≤ i < n : ¬ b[i]) 
       (2) {∃-elimination on (1)} [SOME k] j≤ k < n ∧ ¬ b[k]     // ... so far so good
       
       (3) {∀-elimination on main.A1} [ANY k] 0≤ k < n ∧ b[k]      
           // TWO MISTAKES! (1) ∀-elimination does not produce an [ANY..] flag 
           //               (2) It should be 0≤ k < n ⇒ b[k]  (not a conjunction)
           //                   which then leads to a new challenge: you can't easily get rid of
           //                   the left-side of the ⇒ ( you don't know if 0≤j); to get the contradiction with ¬ b[k] in (1)
       
...
END
